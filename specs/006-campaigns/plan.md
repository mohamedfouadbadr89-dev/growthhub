# Implementation Plan: Phase 6 — Campaigns

**Branch**: `claude/init-growthhub-PaRUm` | **Date**: 2026-04-21 | **Spec**: `specs/006-campaigns/spec.md`

## Summary

Phase 6 adds the Campaigns module: organization-managed campaign records with 30-day aggregated metrics from the Phase 2 sync layer, an AI decisions overlay from Phase 3, AI-generated targeting suggestions (BYOK gate identical to Phase 5), and a push-to-platform mechanism that reuses the Phase 4 `executeAction()` service. No new async jobs are needed — AI suggestions are generated synchronously (single OpenRouter prompt, < 30s).

## Technical Context

**Language/Version**: TypeScript 5.x (backend Hono on Node 20, frontend Next.js 15 App Router)
**Primary Dependencies**: Hono v4, Supabase JS v2, Clerk, Next.js 15, OpenRouter (AI suggestions)
**Storage**: Supabase PostgreSQL — 1 new table: `campaigns`; 2 new rows in `actions_library`
**Testing**: Manual integration testing via quickstart scenarios
**Target Platform**: Railway (backend), Hostinger VPS (frontend)
**Performance Goals**: Campaigns list renders in < 3s; AI suggestions return within 30s
**Constraints**: All tables must have org_id + RLS; BYOK gate applies to AI suggestions; push-to-platform must go through existing `executeAction()` — no direct platform API calls
**Scale/Scope**: Up to 100 campaigns per org; pagination at 50 per page (max 100)

## Implementation Notes

### 1. Database Migration

```sql
-- supabase/migrations/20260420000007_campaigns.sql

CREATE TABLE campaigns (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           TEXT NOT NULL,
  name             TEXT NOT NULL,
  platform         TEXT NOT NULL CHECK (platform IN ('meta','google')),
  status           TEXT NOT NULL DEFAULT 'draft'
                   CHECK (status IN ('draft','active','paused','completed','archived')),
  daily_budget     NUMERIC(14,4),
  targeting        JSONB NOT NULL DEFAULT '{}',
  ad_account_id    UUID REFERENCES ad_accounts(id) ON DELETE SET NULL,
  platform_campaign_id TEXT,
  ai_suggestions   JSONB,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE campaigns ENABLE ROW LEVEL SECURITY;
CREATE POLICY "campaigns_org" ON campaigns
  USING (org_id = auth.jwt()->>'org_id')
  WITH CHECK (org_id = auth.jwt()->>'org_id');

CREATE UNIQUE INDEX idx_campaigns_unique ON campaigns (org_id, name, platform);
CREATE INDEX idx_campaigns_org_id ON campaigns (org_id);
CREATE INDEX idx_campaigns_org_status ON campaigns (org_id, status);
CREATE INDEX idx_campaigns_org_platform ON campaigns (org_id, platform);

-- Trigger: auto-update updated_at
CREATE TRIGGER campaigns_updated_at
  BEFORE UPDATE ON campaigns
  FOR EACH ROW EXECUTE FUNCTION moddatetime(updated_at);
```

### 2. Campaign Service — Metrics Join

```typescript
// backend/src/services/campaigns/campaigns.ts

// List query: left-join campaign_metrics for last 30 days
async function listCampaigns(orgId: string, filters: CampaignFilters) {
  // 1. Query campaigns table with status/platform filters
  // 2. Separately query campaign_metrics last 30 days: GROUP BY campaign_name, platform
  //    SUM(spend), SUM(revenue), SUM(conversions), SUM(impressions),
  //    CASE WHEN SUM(spend)>0 THEN SUM(revenue)/SUM(spend) ELSE 0 END as roas
  // 3. Join metrics to campaigns by (lower(name) = lower(campaign_name) AND lower(platform) = lower(platform))
  // 4. Return campaigns with metrics embedded
}

// Detail query: single campaign + 14-day daily trend + decisions overlay
async function getCampaignById(orgId: string, id: string) {
  // 1. Fetch campaign row
  // 2. Fetch 30-day aggregated metrics (same as list)
  // 3. Fetch 14-day daily trend: SELECT date, SUM(spend), SUM(revenue)/SUM(spend) as roas
  //    FROM campaign_metrics WHERE campaign_name ILIKE campaign.name
  //    AND date >= now()-interval '14 days' GROUP BY date ORDER BY date
  // 4. Fetch decisions: SELECT id, title, confidence_score, status, action_id
  //    FROM decisions WHERE org_id=orgId AND campaign_name ILIKE campaign.name
  //    AND status='active'
  // 5. Return assembled detail object
}
```

### 3. AI Suggestions Service

```typescript
// backend/src/services/campaigns/ai-suggestions.ts

// Reuses resolveApiKey() from creatives service (or a shared helper)
export async function generateAiSuggestions(orgId: string, campaignId: string) {
  // 1. resolveApiKey(orgId) — throws 402 BYOK_REQUIRED if LTD + no key
  // 2. Fetch campaign row (name, platform, daily_budget)
  // 3. Fetch top 3 campaigns by ROAS from campaign_metrics (last 30 days)
  // 4. Build prompt: "Given these top performing campaigns and their ROAS,
  //    suggest targeting for: {name} on {platform} with budget {budget}"
  // 5. POST to OpenRouter, parse JSON response
  // 6. Validate: interests array, age_min, age_max, gender, daily_budget_recommendation
  // 7. UPDATE campaigns SET ai_suggestions = {..., generated_at: now()} WHERE id=campaignId
  // 8. Return suggestions object
}
```

### 7. Frontend API Pattern

All frontend pages use `apiClient` from `@/lib/api-client` with `useAuth()`. Status filter change triggers a new fetch with `?status=<value>` — no page reload. The Create page uses a two-step flow: save campaign first (POST /campaigns → get ID), then call AI suggestions on the returned ID.

### 8. Pagination

```
GET /campaigns?limit=50&offset=0
```
- Default limit: 50
- Maximum limit: 100 (server clamps: `Math.min(limit, 100)`)
- Response always includes `total` count (separate COUNT(*) query or `.count('exact')`)

### 9. Org Isolation Double-Check

Every query in campaigns.ts and ai-suggestions.ts MUST include `.eq('org_id', orgId)` where orgId is extracted from the verified Clerk JWT — never from request body or query params.

### 10. No New Inngest Jobs

AI suggestions are synchronous (target < 30s). Push-to-platform uses the synchronous `executeAction()` which is already tested and doesn't need a background job.
