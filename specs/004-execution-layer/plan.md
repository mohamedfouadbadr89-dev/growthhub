# Implementation Plan: Execution Layer — Actions, Automation & Decision History

**Branch**: `claude/init-growthhub-PaRUm` | **Date**: 2026-04-20 | **Spec**: `specs/004-execution-layer/spec.md`

## Summary

Phase 4 adds the Execution Layer: a system-seeded Actions Library, manual action execution from both the Actions Library and Decision Detail pages, IF→THEN automation rules that fire automatically after each decision generation run, and the **Decision History** table — the constitution-mandated memory and learning loop that captures every execution (manual or automated) with full context.

Platform API write operations are mocked in Phase 4 (registry pattern ensures surgical replacement in Phase 6).

## Technical Context

**Language/Version**: TypeScript 5.x (backend Hono on Node 20, frontend Next.js 15 App Router)
**Primary Dependencies**: Hono v4, Supabase JS v2, Inngest v3, Clerk, Next.js 15
**Storage**: Supabase PostgreSQL — 4 new tables: `actions_library`, `automation_rules`, `automation_runs`, `decision_history`
**Testing**: Manual integration testing via quickstart scenarios
**Target Platform**: Railway (backend), Hostinger VPS (frontend)
**Project Type**: Web application (existing monorepo)
**Performance Goals**: Automation rules fire within 30s of decision generation; history records appear within 5s of execution
**Constraints**: All tables must have org_id + RLS; decision_history schema is constitution-mandated and non-negotiable; actions_library is system-global (no org_id)
**Scale/Scope**: 50–100 automation rules per org; paginated at 50 records per page

## Implementation Notes

### 1. Action Executor Registry Pattern
```typescript
// backend/src/services/execution/action-executor.ts
const ACTION_HANDLERS: Record<string, ActionHandler> = {
  pause_campaign:   async (params, ctx) => ({ success: true, result_data: { simulated: true, ...params } }),
  increase_budget:  async (params, ctx) => ({ success: true, result_data: { simulated: true, ...params } }),
  decrease_budget:  async (params, ctx) => ({ success: true, result_data: { simulated: true, ...params } }),
  send_alert_email: async (params, ctx) => ({ success: true, result_data: { simulated: true, ...params } }),
}

export async function executeAction(templateId, params, orgId, decisionId?) {
  // 1. Fetch template from actions_library
  // 2. Validate required params from parameter_schema
  // 3. Check integration exists for platform (throw INTEGRATION_NOT_CONNECTED if not)
  // 4. Call ACTION_HANDLERS[template.action_type]
  // 5. Insert decision_history record (always, even on failure)
  // 6. Return { historyId, result, resultData }
}
```

### 2. Automation Engine — Rule Matching
```typescript
// backend/src/services/execution/automation-engine.ts
export async function dispatchAutomation(orgId: string, runId: string): Promise<number> {
  // 1. Fetch enabled rules for org
  // 2. Fetch active decisions created in this run (created_at >= run started_at)
  // 3. For each (decision, rule) where rule.trigger_type === decision.type
  //    AND decision.confidence_score >= rule.min_confidence_threshold:
  //    a. executeAction(rule.action_template_id, resolvedParams, orgId, decision.id)
  //    b. Insert automation_run record
  //    c. Update rule.run_count++ and last_fired_at
  // 4. Return total rules executed
}
```

### 4. Action Params Resolution for Auto-Execution
When an automation rule fires, `action_params` may contain `"campaign_id": "auto"` — meaning use the campaign_id from the decision's `data_snapshot`. The automation engine resolves this:
```typescript
const params = { ...rule.action_params }
if (params.campaign_id === 'auto') {
  params.campaign_id = decision.data_snapshot?.campaign_id ?? decision.campaign_id
}
```

### 5. Decision History Construction
**From manual execution** (executeAction called directly):
- `decision` = action template name (if no decision_id) or decision.trigger_condition
- `action_taken` = `"${template.name} — ${Object.entries(params).map(([k,v]) => `${k}: ${v}`).join(', ')}"`
- `trigger_condition` = `"Manual execution"` or `"Manual execution — ${decision.trigger_condition}"`
- `data_used` = `{}` or decision.data_snapshot
- `executed_by` = 'manual'

**From automation** (dispatchAutomation):
- `decision` = decision.trigger_condition
- `trigger_condition` = `"Rule: ${rule.name} — ${decision.trigger_condition}"`
- `data_used` = decision.data_snapshot
- `ai_explanation` = decision.ai_explanation
- `confidence_score` = decision.confidence_score
- `executed_by` = 'automation'

### 7. Decision Detail Page Modification
`app/decisions/[id]/page.tsx` already fetches the full decision. Add an "Execute Recommended Action" button that:
1. Shows a confirmation modal with the recommended_action text and a campaign_id input
2. On confirm: `POST /api/v1/actions/:actionTemplateId/execute` with `{ params, decision_id: id }`
3. The frontend needs to resolve which action template matches the decision type. Use a simple lookup:
   - ROAS_DROP + meta → pause_campaign (meta)
   - SPEND_SPIKE + meta → decrease_budget (meta)
   - SCALING_OPPORTUNITY + meta → increase_budget (meta)

### 9. RLS Notes
- `actions_library`: No RLS needed — use `CREATE POLICY ... USING (true)` for SELECT; no DML for users.
- All other tables: standard `org_id = auth.jwt()->>'org_id'` policy.

### 10. Pre-flight vs Post-flight Logging
- Validation failures (missing params, INTEGRATION_NOT_CONNECTED) → return error, do NOT insert decision_history
- Execution failures (handler throws) → insert decision_history with result = 'failed' and error in result_data
- This distinction keeps history clean (only real execution attempts, not input validation errors)
