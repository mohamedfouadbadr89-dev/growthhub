# Implementation Plan: Phase 1 ‚Äî Foundation

**Branch**: `001-phase-1-foundation` | **Date**: 2026-04-20 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/001-phase-1-foundation/spec.md`

## Summary

Build the end-to-end foundation: Clerk auth (already scaffolded) wired to
auto-organization creation via webhooks, a Hono backend on Railway with JWT
verification middleware and a `/api/v1/` base structure, and a Supabase schema
with RLS-enabled tables (`organizations`, `users`, `subscriptions`,
`audit_logs`). All existing frontend scaffolding is preserved; this phase
completes the missing wiring to make it functional.

## Technical Context

**Language/Version**: TypeScript 5 (frontend + backend)
**Frontend**: Next.js 16 App Router ‚Äî already scaffolded, Clerk already installed
**Backend Framework**: Hono (Node.js) ‚Äî new `backend/` directory in monorepo
**Database**: Supabase PostgreSQL ‚Äî new schema via `supabase/migrations/`
**Auth**: Clerk `@clerk/nextjs` ^7 (already installed) + `@clerk/backend` for backend
**Webhook Verification**: `svix` npm package
**Error Tracking**: `@sentry/node` on backend, `@sentry/nextjs` on frontend
**Testing**: Manual integration tests via curl + Supabase Studio verification
**Target Platform**: Frontend ‚Üí Hostinger VPS; Backend ‚Üí Railway; DB ‚Üí Supabase
**Performance Goals**: Health endpoint < 500ms; sign-up flow < 60s end-to-end
**Constraints**: `service_role_key` backend-only; RLS on all tables; no direct
  Supabase calls from frontend
