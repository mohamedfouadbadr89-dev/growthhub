# Implementation Plan: Phase 2 вЂ” Data Ingestion

**Branch**: `002-data-ingestion` | **Date**: 2026-04-20 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/002-data-ingestion/spec.md`

## Summary

Build the end-to-end data ingestion layer: OAuth connect flows for Meta Ads,
Google Ads, and Shopify; encrypted token storage via Supabase Vault; daily
background sync jobs via Inngest on Railway; and real metrics surfaced on the
Dashboard Overview and Channels pages. Four new tables (`integrations`,
`ad_accounts`, `campaign_metrics` partitioned by date, `sync_logs`) with full
RLS. Manual re-sync trigger exposed via the Integrations page.

## Technical Context

**Language/Version**: TypeScript 5 (frontend + backend)
**Frontend**: Next.js 16 App Router вЂ” existing scaffold; OAuth callback route is new
**Backend Framework**: Hono (Node.js) on Railway вЂ” extends existing Phase 1 backend
**Database**: Supabase PostgreSQL вЂ” 4 new tables via migration
**Auth**: Clerk `@clerk/nextjs` ^7 (existing) вЂ” no changes
**Job Queue**: Inngest вЂ” new `backend/src/jobs/` module; serve at `POST /api/inngest`
**Token Storage**: Supabase Vault (`vault.create_secret()` / `vault.decrypted_secrets`)
**Platform APIs**:
  - Meta Marketing API v21.0 вЂ” `GET /act_{id}/insights`, System User token
  - Google Ads API v19 REST вЂ” `searchStream` + GAQL, developer-token required
  - Shopify Admin GraphQL API вЂ” `orders` query, cursor pagination, `read_orders` scope
**New Backend Dependencies**: `inngest`, `@inngest/hono` (or `inngest/hono` serve)
**New Frontend Dependencies**: none
**Performance Goals**: Manual sync queued < 5s; daily sync < 10min for 500 campaigns; metrics API < 500ms
**Constraints**: All platform tokens in Vault only; `service_role_key` backend-only;
  RLS on all 4 tables; upsert (not insert) for campaign_metrics to ensure idempotency

### Key implementation decisions from research

1. **OAuth state (CSRF)**: Generate a short-lived signed token on `connect/start`,
   store in Supabase `oauth_states` table (or signed JWT вЂ” prefer DB for auditability).
   Validate and consume on `connect/complete`. State expires after 10 minutes.

2. **Token Vault pattern**:
   ```typescript
   // Write
   const { data } = await supabaseAdmin.rpc('vault.create_secret', { secret: refreshToken })
   // data = { id: uuid }  в†’  store in integrations.vault_refresh_token_secret_id

   // Read
   const { data } = await supabaseAdmin
     .from('vault.decrypted_secrets')
     .select('decrypted_secret')
     .eq('id', vaultSecretId)
     .single()
   ```

3. **Inngest registration** in `backend/src/index.ts`:
   ```typescript
   import { serve } from 'inngest/hono'
   import { inngest, functions } from './jobs/inngest.js'
   app.on(['GET', 'PUT', 'POST'], '/api/inngest', serve({ client: inngest, functions }))
   ```

4. **Upsert for idempotent sync** (SC-006):
   ```typescript
   await supabaseAdmin.from('campaign_metrics').upsert(rows, {
     onConflict: 'org_id,ad_account_id,campaign_id,date',
     ignoreDuplicates: false
   })
   ```

5. **Dashboard data wiring**: Dashboard pages use `lib/api-client.ts` (Phase 1)
   with `auth().getToken()` in Server Components. No client-side state needed for
   initial metrics display.
