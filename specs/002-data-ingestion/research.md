# Research: Phase 2 — Data Ingestion

## 1. OAuth Flow Architecture (Frontend/Backend Split)

**Decision**: OAuth redirect URL registered with platforms points to the **Next.js
API route** at `/api/integrations/callback/[platform]` (Hostinger VPS). CSRF state
is generated by the backend, stored in an httpOnly cookie on the Next.js server,
and validated on the callback before forwarding the authorization code to the
backend via a server-to-server POST.

**Rationale**: OAuth providers require a single registered redirect URI. The
Next.js server-side route is the correct boundary — it extracts the code+state
from the browser redirect, validates state (CSRF protection per RFC 6749 §10.12),
then makes a server-side call to the backend (`POST /api/v1/integrations/connect/complete`)
with the code and the user's Clerk Bearer token. The code is never exposed to the
browser, and the token exchange happens entirely server-side on the backend.

**Alternatives considered**:
- Redirect directly to the Hono/Railway endpoint: Works but requires the Railway
  URL to be the registered OAuth redirect, making it harder to change hosting
  without updating every OAuth app registration.
- Client-side state validation: Rejected — RFC 6749 requires server-side
  validation; client-side is insufficient.

## 2. OAuth Token Storage (Supabase Vault)

**Decision**: Store encrypted OAuth refresh tokens (and any access tokens for
short-expiry platforms) in **Supabase Vault** via `vault.create_secret()`. The
`integrations` table stores the returned UUID as `vault_refresh_token_secret_id`.
Tokens are read by the backend via the `vault.decrypted_secrets` view using
`service_role_key`.

```sql
-- Write (from Hono backend with service_role_key)
SELECT vault.create_secret('refresh_token_value') AS secret_id;

-- Read (from Hono backend)
SELECT decrypted_secret FROM vault.decrypted_secrets
  WHERE id = (SELECT vault_refresh_token_secret_id FROM integrations WHERE id = $1);
```

**Rationale**: Supabase Vault uses `pgsodium` for at-rest encryption. Secrets are
identified by UUID, not by name, so the integrations table stores the UUID
reference. Only the backend with `service_role_key` can query
`vault.decrypted_secrets` — the frontend (anon key) cannot. This is the
architecture's intended pattern (CLAUDE.md: "BYOK key stored encrypted in
Supabase Vault").

**Alternatives considered**:
- Store encrypted tokens in an `integrations` column: More portable but requires
  manual key management; Vault is purpose-built and cleaner.
- Secrets referenced by name: Not how Vault works; UUID is the reference mechanism.

## 3. Inngest Setup on Hono (Railway)

**Decision**: Use `serve()` from `inngest/hono` to register all Inngest function
handlers at `POST /api/inngest` on the Hono backend. Daily sync uses Inngest's
cloud scheduler (`triggers: { cron: "0 2 * * *" }` — 2am UTC). Manual sync
triggers use `inngest.send({ name: 'integration/sync.requested', data: { integrationId, orgId } })`
from within a Hono route handler. Both `INNGEST_EVENT_KEY` and
`INNGEST_SIGNING_KEY` are required in production.

```typescript
// Registration
app.on(['GET', 'PUT', 'POST'], '/api/inngest',
  serve({ client: inngest, functions: [dailySyncJob, syncIntegration] })
)

// Cron function
export const dailySyncJob = inngest.createFunction(
  { id: 'daily-sync', triggers: { cron: '0 2 * * *' } },
  async ({ step }) => { /* fan out sync events for each active integration */ }
)

// Manual trigger from route
await inngest.send({ name: 'integration/sync.requested', data: { integrationId } })
```

**Rationale**: Inngest is already in the architecture. Keeping all job logic on
Railway (not Next.js) aligns with the architecture rule that Railway is the
"persistent server for jobs/APIs". The cron runs on Inngest's cloud scheduler —
no additional cron infrastructure needed. `SIGNING_KEY` authenticates requests
FROM Inngest TO our server; `EVENT_KEY` authenticates our server TO Inngest.

**Alternatives considered**:
- pg_cron (Supabase-side): Too tightly coupled to DB; harder to monitor.
- Bull/BullMQ + Redis: More ops overhead; Inngest already chosen.
- Vercel Cron (Next.js): Violates architecture rule — all jobs on Railway.

## 4. Meta Ads API

**Decision**: Use Meta Marketing API **v21.0** (current stable; v20 deprecated
May 2025). Pull campaign insights via `GET /act_{ad_account_id}/insights` with
fields `spend,impressions,clicks,conversions,action_values` and
`date_preset=yesterday` (or `time_range` for backfill). Use a **System User token**
for unattended server-side access — it doesn't expire with user sessions.

**Rationale**: System User tokens are designed for B2B server integrations and
don't tie to individual user login sessions. v21.0 is the latest stable version.
The Insights API is the canonical endpoint for aggregate campaign metrics.

**Alternatives considered**:
- Long-lived User Access Token: Works but expires and needs manual refresh;
  fragile for unattended sync.
- Facebook Node SDK: Adds a dependency; direct HTTP is sufficient for Phase 2.

## 5. Google Ads API

**Decision**: Use Google Ads API **v19** (current) via **REST** `searchStream`
endpoint with GAQL queries. No Node.js client library. OAuth2 scope:
`https://www.googleapis.com/auth/adwords`. A `developer-token` HTTP header is
required on every request (obtained from Google Ads account settings).

```sql
-- GAQL example
SELECT campaign.id, campaign.name, campaign.status,
       metrics.impressions, metrics.clicks, metrics.cost_micros,
       metrics.conversions, metrics.conversions_value,
       segments.date
FROM campaign
WHERE segments.date = '2026-04-19'
```

**Rationale**: REST + GAQL avoids the heavyweight client library (~200MB) and is
fully capable for daily campaign metrics pulls. `cost_micros` divided by 1,000,000
gives spend in the account currency.

**Alternatives considered**:
- Google Ads Node.js client library: Provides TypeScript types but is very heavy;
  not justified for Phase 2's scope.
- gRPC: High-throughput but significantly more complex; REST is sufficient.

## 6. Shopify Admin API

**Decision**: Use Shopify Admin **GraphQL** API (REST deprecated for new apps post
April 2025). OAuth scope: `read_orders`. Pull orders via the `orders` query with
`first: 250, after: endCursor` cursor pagination. Revenue field: `totalPriceSet.shopMoney.amount`.
For Phase 2 (orders only — Shopify has no ad campaigns API).

**Rationale**: Shopify deprecated REST for new public apps in April 2025; GraphQL
is mandatory for new integrations. Cursor-based pagination scales correctly for
bulk historical backfill (30 days) without offset drift.

**Alternatives considered**:
- REST `/orders.json`: Legacy; would require migration later. Rejected.
- Shopify Balance/Payments: Out of scope for Phase 2.

## 7. Dashboard Metrics Aggregation

**Decision**: Backend computes aggregate metrics on-demand per request via SQL
`GROUP BY` on `campaign_metrics`. No materialized views or Redis caching for Phase 2.
The date-partitioned table ensures query planner pruning keeps queries fast at Phase 2
data volumes (< 100k rows).

**Rationale**: Premature optimization is a Phase 3 concern. With indexed queries
on the partitioned table, `GROUP BY org_id` across a 30-day window over a few
thousand rows will comfortably meet the 500ms target.

**Alternatives considered**:
- Pre-aggregate in sync jobs: Denormalized; harder to re-aggregate when the query
  changes. Rejected.
- Materialized views: Added in Phase 3 if benchmarks show the need.
